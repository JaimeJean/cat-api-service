📝 ADR-005 — Estratégia de testes

**Requisitos relacionados:** RNF-02, RNF-03, RNF-06

---

## 🎯 Contexto

O case avalia código e testes com peso de 20%. Preciso definir a estratégia de testes de forma que cada nível cubra um tipo de problema diferente, sem sobreposição.

---

## 💡 Decisão

Quatro níveis de teste, cada um com um propósito e um momento de execução:

| Nível | O que testa | Ferramentas | Onde roda |
| --- | --- | --- | --- |
| Unitário | Lógica de negócio isolada | JUnit 5 + Mockito | CI (todo PR) |
| Integração | Comunicação entre camadas e contratos | JUnit 5 + Testcontainers + MockMvc | CI (todo PR) |
| E2E local (BDD) | Fluxo completo ponta a ponta | JUnit 5 + Docker Compose | Manual no desenvolvimento |
| Smoke test | Validação pós-deploy na AWS | Postman | Manual pós-deploy |

---

## 📌 Justificativa

**Testes unitários (domínio isolado):**

Testam as regras de negócio dentro do domínio da arquitetura hexagonal. Como o domínio não depende de frameworks, banco ou APIs externas (inversão de dependência), consigo testar tudo com **JUnit 5** e **Mockito** mockando as portas de saída. Cobertura mínima de 90% validada pelo JaCoCo, com o build falhando se ficar abaixo. Classes de configuração, DTOs e adapters ficam fora da cobertura porque não contêm lógica de negócio. Esses testes rodam na CI em todo PR, e são rápidos porque não sobem nada externo.

**Testes de integração (camadas conversando):**

Validam que as peças se encaixam: o controller recebe um request, passa pelo use case, chega no repositório e persiste no banco. Pra isso vou usar **Testcontainers** (PostgreSQL real em container) e **MockMvc** pra testar os endpoints. Optei por não usar H2 em memória porque ele tem comportamentos diferentes do PostgreSQL (tipos de dados, upsert, queries específicas), e quero que o teste rode contra o mesmo banco da aplicação. Aqui também valido contratos de API: o endpoint retorna o status correto, o body no formato esperado, e os erros seguindo o padrão de resposta que defini pros endpoints. Também rodam na CI em todo PR.

**Testes E2E local com BDD (Dado/Quando/Então):**

Testam o fluxo completo de ponta a ponta contra o Docker Compose inteiro (aplicação + banco + LocalStack). Seguem a estrutura de BDD:

- **Dado** que a ingestão rodou com sucesso
- **Quando** eu chamo `GET /api/v1/breeds`
- **Então** recebo a lista de raças com suas imagens

Cada cenário tem o resultado esperado documentado. Quando rodo, comparo com o resultado encontrado. Se bate, o cenário passa.

**Smoke test na AWS (validação pós-deploy):**

Depois de subir na AWS, faço uma validação manual: o banco foi populado? Os endpoints respondem? Os dados estão íntegros? Nada se perdeu entre o ambiente local e a cloud? Documento o passo a passo e os resultados no README. Automatizar isso contra a AWS real adicionaria complexidade de gerenciar ambiente, dados de teste e cleanup, o que não compensa no prazo do case. Fica como evolução futura.

---

## 🔄 Alternativas consideradas

**H2 em memória pra testes de integração:** descartei porque tem comportamentos diferentes do PostgreSQL em edge cases (tipos de dados, queries específicas, upsert). Quero que o teste rode contra o mesmo banco que a aplicação usa de verdade. Testcontainers resolve isso.

**Cucumber pra BDD:**  apesar de nunca ter usado a ferramenta eu a conheço, mas pro tamanho do case adiciona complexidade de configuração sem retorno proporcional. Vou escrever os testes E2E com a estrutura Dado/Quando/Então direto no código de teste, sem framework extra. Cucumber faria mais sentido num projeto maior com mais cenários.

---

## ⚖️ Consequências

**Positivas:**

- Cada nível de teste cobre um tipo de problema diferente, sem sobreposição
- Testes unitários rápidos e isolados graças à arquitetura hexagonal
- Testcontainers garante que o banco do teste é igual ao de produção
- BDD documenta os cenários de forma legível (Dado/Quando/Então)
- Smoke test na AWS valida que o deploy não quebrou nada
- JaCoCo enforced no build garante que a cobertura não cai

**Negativas:**

- Testcontainers precisa de Docker rodando pra executar os testes de integração
- Smoke test manual não é automatizado (evolução futura)

---

## 💭 Evolução futura

**Automatização do smoke test:** em cenário produtivo, o smoke test poderia rodar automaticamente após o deploy via GitHub Actions, chamando os endpoints da AWS e validando os responses. Com rollback automático do ECS, se o smoke test falhar, a versão anterior continua rodando.

**Cucumber pra BDD:** se o projeto crescer e tiver mais cenários, Cucumber com feature files em Gherkin ajudaria a organizar melhor e facilitaria leitura dos testes por pessoas não-técnicas.
