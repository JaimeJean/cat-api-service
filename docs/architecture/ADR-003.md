# 📝 ADR-003 → Escolha do mecanismo de fila

**Requisitos relacionados:** RF-08, RF-16, RF-17

---

## 🎯 Contexto

O case pede, como bônus, um fluxo assíncrono onde o usuário informa um e-mail e filtros opcionais (temperamento, origem) e recebe os resultados por e-mail. Pra isso, preciso de um mecanismo de fila que desacople o pedido (receber a requisição e responder “pedido recebido”) do processamento (executar a consulta + enviar o e-mail).

---

## 💡 Decisão

Vou usar **Amazon SQS** como mecanismo de fila, **Amazon SES** para envio de e-mails e **Amazon DynamoDB** para controle de idempotência do consumer. Para execução local, vou usar **LocalStack** no Docker Compose simulando os três serviços.

O fluxo fica: API recebe o pedido → publica na **SQS** → retorna 202 Accepted → consumer consome da fila → verifica idempotência no **DynamoDB** → executa consulta → **SES** envia e-mail → registra mensagem como processada no DynamoDB.

---

## 📌 Justificativa

**Por que SQS:**

Três motivos práticos. Primeiro, custo: o SQS tem 1 milhão de requests grátis por mês no free tier, o que é mais do que suficiente pro case. Segundo, simplicidade: é um serviço gerenciado, não preciso configurar nem manter nada, e a integração com Spring via Spring Cloud AWS é direta. Terceiro, o volume é baixo. Estou enfileirando pedidos de envio de e-mail, não processando milhões de eventos. SQS resolve isso sem complexidade extra.

Vou configurar uma **DLQ (Dead Letter Queue)** no próprio SQS: se o consumer falhar ao processar uma mensagem após X tentativas, ela vai pra uma fila secundária em vez de ficar voltando infinitamente ou se perder. Isso me dá visibilidade sobre falhas e possibilidade de reprocessamento manual ou automatizado.

Outro ponto importante: SQS standard tem entrega “at-least-once”, o que significa que a mesma mensagem pode ser entregue mais de uma vez. Pra controlar isso, optei por usar **DynamoDB** pro registro de mensagens processadas. O consumer faz um PutItem condicional com `attribute_not_exists(messageId)` antes de processar. Se a mensagem já foi processada, o PutItem falha e o consumer descarta. Escolhi DynamoDB em vez de uma tabela no PostgreSQL porque: acesso por chave única (`messageId`) é exatamente o padrão onde DynamoDB faz sentido, tem TTL nativo (limpa registros após 7 dias automaticamente) e funciona mesmo com múltiplas instâncias do consumer (o PutItem condicional é atômico e resolve race conditions).

**Por que não Kafka:**

Kafka resolveria três problemas ao mesmo tempo nesse projeto: fila assíncrona, resiliência na ingestão com retry/DLQ e reprocessamento de falhas. Porém, pro contexto do case, Kafka traz dois problemas reais: custo (o Amazon MSK não entra no free tier) e complexidade local (precisa de Zookeeper + Kafka no Docker Compose, o que pesa o ambiente e dificulta a execução pelo avaliador). Como o único uso de fila no case é o bônus assíncrono, SQS é a escolha que faz mais sentido.

**Por que SES direto (sem SNS):**

No planejamento inicial considerei usar SNS como camada de roteamento entre o processamento e o envio, mas o desacoplamento já está resolvido no código via interface (`NotificationPublisher`). O SES Adapter implementa essa interface diretamente. Se amanhã precisar adicionar Slack ou webhook, posso criar um novo adapter que implementa a mesma interface, sem mexer no domínio. O SNS seria uma camada extra na infra que não agrega valor real ao escopo entregue.

**Por que SES pra e-mail:**

SES é o serviço de e-mail da AWS, se integra bem ao restante da stack e atende diretamente o bônus do case. Para a versão entregue, bastou verificar remetente/destinatários de teste e validar o envio via logs e recebimento real em caixas verificadas.

**Execução local com LocalStack:**

LocalStack no Docker Compose simula SQS, SES e DynamoDB localmente. Spring Profiles cuida de apontar pra LocalStack no profile local e pra AWS real no profile prod. O avaliador roda tudo sem conta AWS.

---

## 🔄 Alternativas consideradas

| Opção | Por que descartei |
| --- | --- |
| Kafka (MSK) | Resolveria mais problemas, mas custo fora do free tier e complexidade local com Zookeeper. Over pro volume do case |
| RabbitMQ | Boa opção, mas adicionaria mais um serviço no Docker Compose sem vantagem sobre SQS que é gerenciado e gratuito na AWS |
| SNS entre consumer e SES | Desacoplamento na infra, mas o desacoplamento já está resolvido no código via interface `NotificationPublisher`. Camada extra sem ganho pro case |
| PostgreSQL pra idempotência | Funcionaria, mas DynamoDB é mais adequado: acesso por chave, TTL nativo, PutItem condicional atômico pra race conditions |

---

## ⚖️ Consequências

**Positivas:**

- Custo baixo / compatível com o free tier
- Sem infra extra pra gerenciar na AWS
- DLQ configurada no SQS (3 tentativas) garante visibilidade sobre falhas
- Idempotência via DynamoDB com PutItem condicional funciona mesmo com múltiplas instâncias
- TTL nativo no DynamoDB limpa registros automaticamente após 7 dias
- Desacoplamento do canal de entrega já resolvido no código via interface
- LocalStack permite execução local completa
- Integração simples com Spring Cloud AWS
- Avaliador consegue rodar tudo sem conta AWS

**Negativas:**

- SQS não resolve resiliência da ingestão da TheCatAPI; isso continua tratado na aplicação
- LocalStack simula o comportamento mas pode ter diferenças sutis em relação aos serviços reais
- DynamoDB é um serviço a mais pra configurar, embora a tabela seja simples

---

## 💭 Em cenário produtivo

Em um sistema real com maior volume e necessidade de resiliência, Kafka com DLQ poderia unificar fila assíncrona, retry mais sofisticado e reprocessamento de falhas em uma infraestrutura única. Para o volume e escopo deste case, SQS + SES + DynamoDB entregam o necessário com menor custo e menor complexidade.
