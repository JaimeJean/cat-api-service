# 📝 ADR-001 → Escolha do banco de dados

**Requisitos relacionados:** RNF-07, RF-07

---

## 🎯 Contexto

Preciso persistir dados de raças de gatos, imagens vinculadas a cada raça e imagens avulsas categorizadas (chapéu, óculos). Imagens de raça têm vínculo com breed (1:N). Imagens de chapéu/óculos são globais (`breed_id` nulo) porque a TheCatAPI não associa categorias com raças. O case sugere DynamoDB ou RDS como opções.

---

## 💡 Decisão

Vou usar **PostgreSQL 16 via Amazon RDS** para produção/cloud e a mesma versão via Docker para execução local. Para gerenciamento de schema, vou usar **Flyway** para controle de migrations, o que garante rastreabilidade e reprodutibilidade de mudanças no banco em qualquer ambiente.

---

## 📌 Justificativa

**Por que relacional e não DynamoDB:**

Os dados têm relacionamento entre si — cada raça pode ter até 3 imagens, e preciso consultar por campos variados (temperamento, origem, raça específica). Num relacional, resolvo isso com tabelas normalizadas, foreign keys e joins. No DynamoDB, teria que desnormalizar, duplicar dados e me limitar a padrões de acesso por chave, o que não faz sentido aqui. Além disso, o DynamoDB brilha quando há necessidade de alta escala horizontal, o que não é o caso — o volume de dados do projeto é pequeno.

**Por que PostgreSQL:**

É open source, tem bom suporte no ecossistema Java/Spring e oferece recursos que ajudam diretamente no problema do case, como upsert com `ON CONFLICT DO UPDATE`, útil para ingestão idempotente. Como a ingestão roda no startup, a aplicação pode reiniciar e tentar persistir dados novamente. Com upsert, o banco resolve isso de forma atômica — se o registro já existe, atualiza; se não existe, insere.

**Sobre migrations com Flyway:**

Como estou usando banco relacional, preciso de controle de schema. Optei pelo Flyway porque é simples, se integra bem com Spring Boot e deixa as migrations versionadas no repositório. Isso facilita tanto a evolução do schema quanto a reprodução do ambiente por qualquer pessoa que clone o projeto.

**Custo:**

Para o case, a instância usada no RDS foi mantida enxuta, com foco em custo controlado e possibilidade de destruição do ambiente após os testes. A estratégia de FinOps não depende só da escolha do banco, mas também do uso efêmero da infraestrutura.

---

## 🔄 Alternativas consideradas

| Opção | Por que descartei |
| --- | --- |
| DynamoDB | Relacionamentos seriam forçados, padrões de consulta variados não combinam com acesso por chave |
| MySQL | Também atenderia, mas PostgreSQL oferece recursos e sintaxe mais convenientes pro cenário escolhido |

---

## ⚖️ Consequências

**Positivas:**

- Modelo normalizado com integridade referencial
- Upsert nativo pra idempotência
- Migrations versionadas com Flyway
- Facilidade de execução local via Docker
- Compatível com o volume e a complexidade do domínio do case

**Negativas:**

- Precisa de migrations pra gerenciar schema — resolvido com Flyway
- Não escala horizontalmente como DynamoDB — irrelevante pro volume do case

---

## 💭 Nota: DynamoDB no projeto

DynamoDB foi descartado como banco de dados principal da aplicação, mas foi mantido para um uso pontual onde faz sentido: controle de idempotência do consumer assíncrono, com chave `messageId` e TTL de 7 dias. Esse é um cenário de acesso por chave simples, sem relacionamentos, que combina bem com DynamoDB e não contradiz a decisão de usar PostgreSQL para o domínio principal da aplicação.
