# 📝 ADR-004 → Estratégia de deploy

**Requisitos relacionados:** RNF-15, RNF-17, RNF-18

---

## 🎯 Contexto

Preciso definir como a aplicação vai rodar localmente (pra desenvolvimento e avaliação) e como vai ser provisionada e implantada na AWS. O case avalia “Facilidade de deploy” com peso de 20%, então preciso garantir que qualquer pessoa consiga subir o projeto facilmente, com infraestrutura reproduzível e com custo controlado.

---

## 💡 Decisão

**Local:** Docker Compose orquestrando a aplicação, PostgreSQL e LocalStack (**SQS, SES e DynamoDB**).

**Infraestrutura na AWS:** provisionada via **Terraform** com state remoto no **S3** e locking via **`use_lockfile = true`** no backend.

**Deploy na AWS:** imagem Docker publicada manualmente no **ECR**, executando no **ECS Fargate** atrás de um **ALB (Application Load Balancer)**.

**Estratégia operacional da entrega:**

- infraestrutura provisionada manualmente via Terraform;
- imagem Docker buildada e publicada manualmente no ECR;
- atualização do serviço ECS via nova task definition / novo deploy após push da imagem.

**Segurança:** IAM com **least privilege** (roles separadas pra Terraform, ECS e aplicação) e **Secrets Manager** pra credenciais sensíveis (RDS, TheCatAPI key).

---

## 📌 Justificativa

**Por que Docker Compose localmente:**

Um único `docker-compose up` sobe tudo: aplicação, banco e LocalStack simulando os serviços AWS usados no projeto. Quem for avaliar não precisa de nada além de Docker instalado.

**Por que Terraform:**

Com Terraform, toda a infraestrutura fica como código versionado no repositório. Qualquer pessoa pode rodar `terraform apply` e ter o ambiente equivalente. E o mais importante pro case: `terraform destroy` derruba tudo quando não preciso mais, o que se alinha com a estratégia de FinOps (RNF-17).

**Por que S3 com lockfile pro state do Terraform:**

O state do Terraform não pode ficar só na minha máquina, então ele fica armazenado no S3. Para evitar conflito entre execuções simultâneas, utilizei o mecanismo mais atual do backend S3 com `use_lockfile = true`, sem necessidade de tabela DynamoDB dedicada só para locking.

**Por que ECS Fargate e não EC2 ou Lambda:**

EC2 exigiria gerenciar instâncias, patching e scaling manual. Isso aumenta a responsabilidade operacional sem ganho real pro case. Lambda também não foi adotado: ele serve melhor para funções pontuais disparadas por eventos, não para uma API REST completa em Spring Boot. Além do cold start, a experiência operacional para esse tipo de aplicação seria pior do que rodar o container continuamente no Fargate.

**Por que ALB na frente do Fargate:**

O ALB recebe as requisições HTTP e roteia pro container. Também faz o health check (`/actuator/health`) pra garantir que só containers saudáveis recebam tráfego. O Fargate roda em **subnet pública** com IP público atribuído automaticamente. O correto pra produção seria subnet privada + NAT Gateway, mas o NAT Gateway traria custo fixo desnecessário pro case. Não estamos lidando com uma carga de produção real nem com um cenário multi-serviço sensível o suficiente para justificar esse custo.

**Security Groups (quem fala com quem):**

A comunicação entre os componentes é controlada por Security Groups com regras mínimas:

- **ALB:** aceita tráfego externo nas portas 80/443
- **Fargate:** aceita tráfego somente vindo do ALB (porta da aplicação)
- **RDS:** aceita conexão somente vinda do Fargate (porta 5432)

Cada componente só é acessível por quem precisa.

**Por que ECR:**

O ECR é o registry de imagens Docker da AWS e se integra naturalmente com o ECS. Para o case, o fluxo manual de build + push foi suficiente e mais pragmático do que montar uma esteira CI/CD completa dentro do prazo.

**Por que o deploy ficou manual (sem GitHub Actions):**

Inicialmente considerei automatizar build, push e deploy com GitHub Actions, mas decidi priorizar a conclusão funcional do case dentro do prazo. A solução entregue mantém a infraestrutura reproduzível em Terraform e o deploy da imagem é executado manualmente no ECR, o que reduz complexidade sem comprometer a validação técnica da solução.

**Estratégia de rollback:**

Se o container novo não passar no health check (`/actuator/health`), o ECS mantém a versão anterior rodando e marca o deploy como falho. Não preciso implementar nada extra, já vem de graça com o ECS.

**Estratégia de segurança (IAM + Secrets Manager):**

Separei em três camadas:

*IAM pro Terraform:* um usuário (ou role) com permissões amplas pra provisionar infra (criar VPC, RDS, ECS etc.). Esse acesso é usado só durante o provisionamento.

*IAM pro ECS Fargate:* duas roles separadas seguindo o princípio de least privilege. A **Task Execution Role** é usada pelo ECS pra puxar a imagem do ECR e enviar logs pro CloudWatch. A **Task Role** é usada pela aplicação em runtime pra acessar SQS, SES, DynamoDB e Secrets Manager.

*Secrets Manager pra credenciais:* credenciais do RDS (senha master gerenciada pela própria AWS) e a API key da TheCatAPI ficam no Secrets Manager. O ECS injeta os valores no container como variáveis de ambiente, e o Spring consome isso normalmente.

## 🔄 Alternativas consideradas

| Opção | Por que descartei |
| --- | --- |
| EC2 | Gerenciamento de instâncias, patching e scaling manual. Mais complexidade operacional sem ganho |
| Lambda | Serve pra gatilhos e funções pontuais, não pra rodar API REST completa em Spring Boot |
| CloudFormation | Terraform tem sintaxe/ecossistema mais familiar e foi a ferramenta estudada para o case |
| GitHub Actions no escopo final | Seria útil, mas aumentaria o escopo de automação além do necessário para concluir a entrega funcional dentro do prazo |

---

## ⚖️ Consequências

**Positivas:**

- `docker-compose up` sobe tudo localmente, sem dependências externas
- Infraestrutura versionada e reproduzível via Terraform
- `terraform destroy` derruba tudo, alinhado com FinOps
- Deploy funcional na AWS com ECS Fargate + ALB + ECR
- Rollback automático do ECS se o health check falhar
- Fargate elimina gerenciamento de instâncias
- Security Groups restringem comunicação ao mínimo necessário entre componentes
- Credenciais nunca ficam expostas em código ou em arquivo versionado
- Cada componente tem só as permissões que precisa (least privilege)
- Avaliador pode rodar local sem conta AWS ou subir a infra na AWS com Terraform

**Negativas:**

- O build/push da imagem ficou manual, então não há esteira CI/CD completa nesta versão
- ALB tem custo fixo, embora o ambiente seja efêmero e possa ser destruído após a demonstração
- Secrets Manager adiciona pequeno custo mensal por secret, ainda aceitável para o case

## 🏗️ Infraestrutura completa no Terraform

| Recurso | Finalidade |
| --- | --- |
| VPC + Subnets públicas | Isolamento de rede para ALB e ECS |
| Subnets privadas de banco | Isolamento do RDS |
| Security Groups | Controle de acesso entre componentes |
| ALB | Ponto de entrada público, roteamento pro Fargate, health check |
| ECR | Registry de imagens Docker |
| ECS Cluster + Service + Task Definition | Execução do container no Fargate |
| RDS (PostgreSQL 16) | Banco de dados da aplicação |
| SQS + DLQ | Fila do fluxo assíncrono |
| SES | Envio de e-mails |
| DynamoDB | Idempotência do consumer assíncrono |
| CloudWatch | Logs, métricas, alarmes e inspeção operacional |
| IAM (Terraform) | Permissões pra provisionamento de infra |
| IAM (Task Execution Role) | Permissões pro ECS puxar imagem do ECR e enviar logs |
| IAM (Task Role) | Permissões da aplicação em runtime (SQS, SES, DynamoDB, Secrets Manager) |
| Secrets Manager | Credenciais do RDS e API key da TheCatAPI |
| S3 | Backend remoto do state do Terraform |

## 💭 Evolução futura

**Pipeline CI/CD:** automatizar build, testes, push no ECR e atualização do ECS com GitHub Actions.

**Subnet privada + NAT Gateway:** em ambiente produtivo com dados sensíveis ou múltiplos serviços, o Fargate rodaria em subnet privada com NAT Gateway. Pro case, o custo não se justifica.

**Route 53 (DNS):** hoje a API é acessada pelo DNS do ALB. Com Route 53, daria pra ter um domínio amigável apontando pro ALB.
